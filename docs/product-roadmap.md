# 产品定位与阶段路线

## 项目定义

`vsummary` 不是一个单纯的“视频转文档脚本”，而是一个**本地优先、支持 BYOK 的交互式音视频学习工具**。

它的核心目标是把音视频内容转成可阅读、可追问、可继续组织的知识结构，而不是只生成一次性摘要后结束。

可以把它理解为：

- 长期愿景：`本地版 NotebookLM for video`
- 第一阶段交付：`稳定的视频转写与结构化总结工具`


## 核心愿景

用户上传一段视频后，系统不仅能给出摘要，还能沉淀出一套可探索的知识材料：

- 带时间戳的转写文本
- 分章节摘要
- 多层级思维导图
- 基于内容的问答入口
- 后续可扩展为多视频联合检索与学习

这个方向更接近“个人音视频知识工作台”，而不是单次处理器。

## Phase 1

### 目标

先跑通“视频 -> 转写 -> 总结 -> 导图”的稳定闭环，解决有无问题。

### 范围

- 上传本地视频或音频
- 使用 `whisper.cpp` 完成转写
- 生成分段摘要与总体摘要
- 生成多层级思维导图数据
- 导出 Markdown / JSON / Mermaid

### 结果标准

- 用户能够稳定得到一份结构化总结
- 输出结果可复用，而不是一次性文本
- 为后续问答和检索保留统一数据结构

## Phase 2

### 目标

引入交互式学习能力，建立项目真正的护城河。

### 范围

- 视频内容问答
- 章节与导图节点联动
- 基于时间戳回溯原视频片段
- 本地检索索引与上下文召回
- 多轮会话与学习记录

### 关键能力

- transcript chunking
- 时间戳索引
- 轻量 RAG
- 会话状态管理
- 节点级上下文追问

## 当前产品判断

现阶段最合适的项目表述是：

**本地优先的交互式视频学习与总结工具**

而不是：

- 批量媒体处理器
- 单纯的视频转文档工具
- 一次性摘要脚本

## 技术路线建议

### 当前阶段

优先保证核心流程清晰：

- 媒体处理
- 转写
- 切块
- 总结
- 导图生成

UI 可以先服务于验证流程，不急着一次做到最终形态。

### 框架建议
- `Streamlit`：适合工具台式原型，但不建议把核心逻辑绑死在页面回调里

长期更稳的方向：

- `FastAPI + React`

原因不是现在就要重投入前端，而是后续的导图交互、会话状态、知识检索会越来越像一个真正的应用，而不是一个脚本外壳。

## 非目标

当前阶段不追求：

- 大而全的插件体系
- 提前抽象复杂 Agent 框架
- 先做多平台桌面端发布
- 先做重度协作能力

先把核心体验做对，比提前铺很多壳更重要。
