# Series Scope RAG INTENT

## 1. 文档目的

本文只回答业务与产品意图，不讨论具体代码怎么改。

它要回答的是：

1. 这次重构到底要解决什么问题
2. 为什么现在必须做
3. 最终用户在什么场景下受益
4. 最高层的业务验收标准是什么

---

## 2. 业务背景

当前 `series scope` 的问答链路已经出现明显产品问题：

1. 首问很慢，系列问题常常在真正回答前先做大量初始化工作
2. 回答偏短、偏保守，用户感知像“模型爱答不理”
3. 很多问题只消费 summary，高层概况能回答，但细节和证据深度不够
4. planner / retrieval / answer 的职责重叠，导致链路复杂但收益不明显

这些问题不是单点 bug，而是当前 `series scope` 架构方向本身不够合适。

---

## 3. Business Goal

将 `series scope` 从“重 planner、多节点执行图”的链路，重构为“轻量查询理解 + 统一检索 + 基于证据回答”的系列问答能力，使系列问题能够更快、更稳、更自然地回答。

一句话版本：

> 让用户在系列维度提问时，系统能稳定地先理解问题，再统一取证，再自然回答，而不是先跑一套复杂编排。

---

## 4. 目标用户

### 4.1 直接用户

视频知识工作台中的最终使用者，典型行为包括：

1. 对整个系列发问
2. 想先了解系列整体内容
3. 想查某个概念、某个框架、某节视频讲过什么
4. 想定位某个术语、时间点、流程或原话

### 4.2 间接受益者

后续维护该 Agent 链路的开发者。

他们需要：

1. 更简单的心智模型
2. 可调试的中间产物
3. 更稳定的提示词与输出合同
4. 更容易验收和回归的结构

---

## 5. 非目标

这次重构不以以下内容为目标：

1. 同时重做 `video scope`
2. 同时重做 notes / cards / mindmap 的完整生成体系
3. 做成通用多知识库 router 平台
4. 一次性引入大量新基础设施

本次范围只聚焦：

1. `series scope` 的查询理解
2. `series scope` 的统一检索
3. `series scope` 的回答合成
4. 调试与验收能力

---

## 6. 核心用户流

下面的用户流只描述目标行为，不描述具体类名和代码实现。

### 6.1 系列总览类问题

```text
用户进入系列主页
-> 用户提问：“这个系列讲了啥”
-> 系统识别为 overview 问题
-> 系统优先命中 series 级高层摘要
-> 系统补充相关视频 summary 证据
-> 系统返回：系列总览 + 主题分组 + 学习顺序
```

### 6.2 概念解释 / 比较类问题

```text
用户提问：“Copilot 模式是啥” 或 “JManus 和 AgentScope 有什么区别”
-> 系统识别为 concept / compare 问题
-> 系统先用高层摘要定位相关视频
-> 系统再补 transcript 级细节证据
-> 系统返回：定义 / 上下文 / 区别
```

### 6.3 定位类问题

```text
用户提问：“哪一节讲过 Nacos 3”
-> 系统识别为 locate 问题
-> 系统优先命中 chapter summary 和 transcript chunk
-> 系统返回：视频标题 + 大致位置 + 原意说明
```

---

## 7. 核心业务判断

本次重构有 4 个关键产品判断：

1. overview 问题应该有 series 级高层语料，不应总靠“遍历所有视频 summary 再临时总结”
2. summary 和 transcript 应属于同一个统一检索体系，不应被流程性拆开
3. prompt 的职责应当更轻，只负责“帮助检索”和“基于证据回答”，不负责复杂执行编排
4. 验收不能只看最终回答，而必须看中间证据产物是否合理

---

## 8. 核心验收标准

这里只写业务验收，不写实现细节。

### 8.1 系列总览类

1. 对“这个系列讲了啥”这类问题，系统必须能稳定回答出系列主线，而不是只给零碎 summary 改写
2. 回答中应能体现主题分组或学习脉络，而不只是列视频名

### 8.2 概念 / 比较类

1. 对“Copilot 模式是啥”“JManus 和 AgentScope 有什么区别”这类问题，回答不能只停留在一句概念定义
2. 系统应能结合系列上下文回答，而不是像脱离课程语境的百科句子

### 8.3 定位类

1. 对“哪一节讲过 Nacos 3”这类问题，系统必须给出视频或章节级定位结果
2. 回答应包含大致位置或章节含义，不能只说“讲过”

### 8.4 性能与可用性

1. 系列首问不能再把大规模索引构建成本直接压到用户交互链路上
2. 即使 series synopsis 尚未生成或正在重建，系列问答也必须保持可用

### 8.5 稳定性

1. 新链路必须比旧链路更容易调试
2. 必须能输出中间产物，让开发者判断系统为什么这样回答

---

## 9. 审核重点

在你审核这份 INTENT 时，建议重点看下面 4 件事：

1. 业务目标有没有偏
2. 用户流是不是覆盖了你真正关心的三类问题
3. 非目标范围有没有划清
4. 核心验收标准是否符合你对产品结果的预期

如果这四件事成立，后续 SPEC 才有意义。
