# Agent 用户模拟测试文档

本文从真实用户视角出发，给出一组可直接照着操作的手动测试用例，用来验证当前 Studio 中的 Agent 对话、事件流、工具链与最终回答是否符合预期。

注意：

- 这组测试属于低优先级的真实模型对话回归
- 默认不纳入日常后端测试与提交前检查
- 只有在你明确想验 Agent 实际回复质量时，才手动运行对应脚本
- 项目脚本要求显式追加 `--manual` 才会真正执行

本文重点验证：

- `首页` 是否只保留工作区浏览，不再显示 AI 对话
- `思路块` 是否出现且内容合理
- `工具链块` 是否按顺序展示真实工具名
- `最终回答` 是否与工具结果一致
- 系列级问题是否真的先走系列候选缓冲区，再进入视频级核验，而不是模型硬猜

## 1. 测试前提

开始前请确认：

1. 前端与后端都已启动
2. 已配置可用的 OpenAI 兼容模型
3. 至少存在一个系列，例如 `Agent Frameworks`
4. 该系列下至少有 2 到 4 个视频
5. 至少 1 个视频已经生成过 `summary`
6. 最好同时准备：
   - 一个已有 summary 的视频
   - 一个没有 summary 的视频

推荐样例数据：

- `1-4 准备工作：百度地图API秘钥(AK)`：有 summary
- `1-5 准备工作：安装Nacos 3`：有 summary
- `1-6 仿Manus能自主决策的框架：Jmanus`：有 summary
- `1-7 具备ReAct核心能力的框架：AgentScope`：无 summary 或待生成

## 2. 统一观察点

每条测试都至少观察下面四项：

1. 是否出现 `思路` 折叠块
2. 是否出现 `工具链` 折叠块
3. 展开后是否能看到真实工具名，例如 `list_series_videos`
4. 最终回答是否只引用了真实工具结果，没有幻觉参数或内部技术词

首页额外关注：

- 工作区首页不应再显示聊天面板
- 只有进入 `series` 或 `video` 上下文后，左侧才出现 AI 对话

额外关注：

- 不应再出现 `__FROM_LIST__`、`*pending_from_*` 这类伪参数
- 不应把 Planner 的内部理由直接挂在最终回答下面
- 最终回答应显示耗时

## 3. 系列级模拟测试

### 3.1 询问系列主题

用户场景：

- 进入某个系列首页
- 不选具体视频
- 在聊天框输入：

```text
这个系列主要讲了哪些主题？
```

预期思路：

- Agent 应识别为 `series` 级问题
- 不应只围绕当前某一个视频回答
- 如果信息不足，应先读取系列视频列表并维护候选缓冲区，再按需读取候选视频摘要

预期事件流：

1. 出现 `正在思考`
2. 思路完成后，显示系列级归纳意图
3. 出现工具链块
4. 展开工具链，至少应看到：

```text
list_series_videos
```

常见情况下接下来还应看到：

```text
add_series_candidates
```

进入候选核验阶段后，通常还会看到：

```text
get_video_summary
```

预期回答：

- 回答应是系列级主题归纳
- 应能自然提到“准备工作”“框架介绍”“自主决策”“ReAct”等主题
- 如果某个视频没有摘要，允许自然说明“部分视频还没有可用摘要”

失败信号：

- 工具链里没有 `list_series_videos`
- 还没建立候选缓冲区就直接去读 `get_video_summary` 或 `get_video_transcript`
- 工具链里出现假参数，如 `__FROM_LIST__`
- 回答只总结了一个视频
- 回答里出现 `tool_calls`、`payload`、`schema` 等内部术语

### 3.2 询问系列学习路径

用户输入：

```text
这个系列适合怎么学？
```

预期事件流：

- 应先出现思路块
- 常见工具链：
  - `list_series_videos`
  - `add_series_candidates`
  - `get_video_summary` 若干次

预期回答：

- 应输出一条学习顺序建议
- 不应只复制视频标题
- 如果只有标题没有摘要，允许回答退回到“基于标题的初步学习路径”

失败信号：

- 完全不使用任何系列级工具
- 回答伪装成知道每期详细内容，但实际没有摘要支撑

### 3.3 询问系列里某两个视频的关系

用户输入：

```text
Jmanus 和 AgentScope 在这个系列里分别承担什么角色？
```

预期事件流：

- `list_series_videos`
- `add_series_candidates`
- 如有可用摘要，按需 `get_video_summary`

预期回答：

- 应体现“系列内部角色对比”
- 应能区分：
  - `Jmanus` 更偏自主决策
  - `AgentScope` 更偏 ReAct

失败信号：

- 回答把两个视频混成一回事
- 明显编造未从标题或摘要得出的细节

## 4. 视频级模拟测试

### 4.1 询问某知识点在视频哪里出现

用户场景：

- 进入某个具体视频
- 输入：

```text
百度地图 API Key 在视频什么位置提到的？
```

预期事件流：

1. `思路块`
2. `工具链块`
3. 展开工具链应看到：

```text
get_video_transcript
video_seek
```

预期界面行为：

- 右侧切到视频预览
- 预览自动跳到对应时间点

预期回答：

- 自然说明大概时间位置
- 不应只跳转不解释

失败信号：

- 没有 `get_video_transcript`
- 视频没有跳转
- 时间点和回答不一致

### 4.2 直接请求打开知识卡片

用户输入：

```text
打开知识卡片
```

预期事件流：

- 应出现思路块
- 工具链里应看到：

```text
open_knowledge_cards
```

预期界面行为：

- 右侧切到知识卡片页

预期回答：

- 自然说明已经打开知识卡片

### 4.3 请求帮我记重点

用户输入：

```text
帮我记一下这个视频的重点
```

预期事件流：

- 工具链中应看到：

```text
save_note
```

如果实现上会顺手切页，也可能看到：

```text
open_notes
```

预期界面行为：

- 新笔记落到笔记页数据里
- 笔记来源应是 `agent`

预期回答：

- 应明确说“已经帮你记下”

失败信号：

- 回复说记好了，但笔记列表没有新增
- 新增笔记为空标题或空内容

## 5. 生成类模拟测试

### 5.1 没有导图时请求打开导图

前提：

- 当前视频已有 summary
- 当前视频没有 mindmap

用户输入：

```text
打开思维导图
```

预期事件流：

- 工具链应看到：

```text

open_mindmap
```

预期界面行为：

- 前端触发导图生成
- 完成后停留在思维导图页

失败信号：

- 只打开，不生成
- 只生成，不打开

### 5.2 没有概况时请求打开概况

前提：

- 当前视频没有 summary

用户输入：

```text

```

预期事件流：

- 工具链应看到：

```text
generate_overview

```

预期界面行为：

- 触发 summary 生成
- 成功后停留在概况页

## 6. 事件流交互专项检查

这组测试专门看“显示方式是否像一个真正的事件流”。

### 6.1 思路块显示检查

任选一个问题，例如：

```text
这个系列主要讲了哪些主题？
```

检查点：

1. 是否先显示 `正在思考`
2. 思考完成后是否收敛成一个可折叠思路块
3. 思路块是否带耗时
4. 思路块内容是否是“对用户可见的思路摘要”，而不是内部技术碎片

失败信号：

- 思路内容直接暴露内部规划术语
- 思路块根本不出现
- 思路完成后又被最终回答覆盖掉

### 6.2 工具链顺序检查

仍以系列问题为例，展开工具链后检查：

1. 步骤是否有先后顺序
2. 是否显示真实工具名
3. 是否显示状态：
   - `进行中`
   - `已完成`
4. 是否显示每步耗时

失败信号：

- 只有中文说明，没有真实工具名
- 步骤顺序颠倒
- 结果块显示的不是执行后的真实参数

### 6.3 最终回答流式输出检查

用户输入：

```text
总结当前系列
```

检查点：

1. 回答是否逐步出现，而不是整段瞬间出现
2. 回答完成后是否显示耗时
3. 最终回答是否晚于思路块与工具链块出现

失败信号：

- 最终回答比工具链先出现
- 回答一次性整段弹出

## 7. 异常场景模拟测试

### 7.1 提出与当前知识库无关的问题

用户输入：

```text
帮我写一份旅游攻略
```

预期：

- 应礼貌拒绝或说明超出当前工作台范围
- 不应调用工作区工具

## 8. 推荐手测顺序

建议按下面顺序手测，最容易定位问题：

1. `打开知识卡片`
2. `帮我记一下这个视频的重点`
3. `百度地图 API Key 在视频什么位置提到的？`
4. `这个系列主要讲了哪些主题？`
5. `这个系列适合怎么学？`
6. `Jmanus 和 AgentScope 在这个系列里分别承担什么角色？`

## 9. 通过标准

本轮 Agent 工具链体验可以视为通过的最低标准：

1. 系列级问题会先触发系列候选缓冲，再进入视频级核验，而不是只猜标题
2. 工具链展示真实工具名，而不是只有中文标签
3. 不再出现 `__FROM_LIST__`、`*pending_from_*` 之类伪参数
4. 思路、工具链、回答三段顺序正确
5. 最终回答与工具结果一致，不编造不存在的摘要内容

## 10. 上下文预算专项测试

这组测试验证当前新加的“上下文预算可视化”是否可信、是否会跟随当前 scope 正确变化。

### 10.1 在首页 / series / video 三种界面之间切换

操作步骤：

1. 先停留在工作区首页
2. 确认首页没有聊天面板，也没有 `上下文预算`
3. 进入某个系列首页
4. 观察聊天面板顶部的 `上下文预算`
5. 进入某个具体视频
6. 再观察一次

预期结果：

- 首页不显示聊天面板，也不显示 `上下文预算`
- 进入 `series` 与 `video` 后都能看到 `上下文预算`
- 预算条不会消失或报错
- `工作区上下文` 一项在 video 级通常会高于 series 级
- 当前 scope 切换后，顶部预算应随之变化

失败信号：

- 首页仍然显示聊天面板
- 进入某个 series 或 video 后预算条消失
- 预算一直显示同一个值，和当前上下文无关
- 出现 `NaN`、空白百分比或负数

### 10.2 在连续对话后观察最近消息占比

操作步骤：

1. 在同一个视频下连续发 3 到 5 条普通问题
2. 每问完一条都观察预算条中的来源拆分

推荐问题：

```text
这个视频主要讲了什么？
```

```text
它的核心问题是什么？
```

```text
帮我用更通俗的话重述一下
```

预期结果：

- `最近消息` 这一项会逐步上升
- `系统指令` 基本稳定
- 总预算会有小幅增加，而不是完全不动

失败信号：

- 连续对话后预算完全不变
- 最近消息永远是 0
- 对话一多就直接报错，没有任何预算预警

## 11. 会话恢复专项测试

这组测试验证“刷新页面后，对话正文能否恢复”。

注意：当前阶段恢复的是用户与助手正文，不要求恢复思路块、工具链块、流式中间态。

### 11.1 刷新后恢复当前 video 对话

操作步骤：

1. 进入某个具体视频
2. 连续发两条问题，例如：

```text
这个视频主要讲了什么？
```

```text
帮我记一下这个视频重点
```

3. 确认聊天区已经出现这两轮正文
4. 直接刷新浏览器
5. 回到同一个视频同一个工具页

预期结果：

- 刷新后能看到刚刚的用户消息和助手消息
- 恢复出来的消息顺序正确
- 恢复的是正文，不要求恢复思路块和工具链块
- 不应重复插入多份相同历史

失败信号：

- 刷新后聊天全丢
- 同一条历史出现两遍或多遍
- 恢复成了别的 scope 的聊天记录

### 11.2 切换到别的 scope，再切回原 scope

操作步骤：

1. 在某个视频里完成一轮对话
2. 切到 series 首页
3. 再切回工作区首页
4. 最后切回原视频

预期结果：

- 原视频 scope 的历史仍存在
- series 应保持自己的聊天线程
- 工作区首页不参与聊天线程，也不应覆盖 video 的恢复记录

失败信号：

- 切换回来后历史被欢迎语重置
- series 的聊天跑进了 video
- 回到首页时又错误展示了某个旧聊天线程

### 11.3 重启前端后恢复

操作步骤：

1. 先完成一轮真实对话
2. 关闭前端 dev server
3. 重新启动前端
4. 打开相同页面并回到相同 scope

预期结果：

- 只要后端还在，之前的正文仍可恢复
- 不要求恢复输入框草稿
- 不要求恢复流式进行中的中间状态

失败信号：

- 重启前端后历史全部消失
- 恢复接口报 500 或前端空白

## 12. 轻量压缩入口专项测试

这组测试目标不是验证“摘要质量有多高”，而是验证长对话不会无限膨胀，以及系统在长会话下仍能继续工作。

注意：当前这一步的压缩作用于 Agent 运行时记忆，用户可见 transcript 不会被直接替换掉。

### 12.1 连续追问制造长对话

操作步骤：

1. 进入某个具体视频或系列
2. 连续追问 10 到 15 轮
3. 每轮尽量让输入长一点，示例如下：

```text
请结合你刚刚的回答，再更详细地解释一遍，并且保留前面提到的关键点、关系和学习顺序建议。
```

也可以多次复制这一句来拉长上下文。

预期结果：

- 对话在长轮次下仍能继续
- 不会在较短轮次就因为上下文过长直接崩掉
- 顶部 `上下文预算` 会持续上涨
- 即使进入长对话，后续回答仍应基本保持与当前主题相关

失败信号：

- 长对话后突然完全失忆，跟当前话题断开
- 还没到很长轮次就直接报上下文错误
- 预算没有变化，但回答明显已经受历史影响

### 12.2 压缩后继续追问

操作步骤：

1. 先完成一长串对话
2. 再问一个依赖前文的问题，例如：

```text
基于我们刚刚前面聊过的内容，帮我再压缩成三条主线。
```

预期结果：

- 回答仍能延续前文主题
- 不应完全忘掉刚才的对话方向
- 即使细节略有收敛，也不应答非所问

失败信号：

- 回答像是全新会话，完全不承接上文
- 直接说“不知道前文”
- 出现明显与当前对话无关的跳题

## 13. 推荐完整手测顺序

如果你想一次把新老能力都过一遍，建议按下面顺序：

1. `这个系列主要讲了哪些主题？`
2. `Jmanus 和 AgentScope 在这个系列里分别承担什么角色？`
3. 进入具体视频后问：`百度地图 API Key 在视频什么位置提到的？`
4. 再问：`帮我记一下这个视频的重点`
5. 观察顶部 `上下文预算` 是否变化
6. 刷新页面，确认当前 scope 的正文是否恢复
7. 连续追问 10 到 15 轮，观察长对话是否还能继续

## 14. 本轮新增能力的通过标准

这一轮如果要判定通过，除了前面的工具链标准外，至少还应满足：

1. `上下文预算` 会随 scope 和对话轮次变化
2. 刷新页面后，当前 scope 的聊天正文可以恢复
3. 不同 scope 之间的恢复记录互不串线
4. 长对话下系统仍可继续工作，不会很早就因上下文膨胀崩掉
5. 轻量压缩即使发生，也不应把用户可见聊天记录直接改写掉
